# Vision — A Company Brain for SMBs

## 出発点：補助金は最も定型化された知的労働

日本の中小企業向け補助金は、典型的な「人月で稼ぐコンサルが中抜きしている
市場」の一例。

- 公募要領（30〜60ページの PDF）を読む
- 事業者へヒアリングする
- 経費計画を組む
- 申請書を書く
- 採択基準と照らして自己採点する
- 不採択時に再申請する

これらはすべて**ルールが明文化されている定型作業**である。一方で報酬は
「成功報酬 10〜20%」で、補助金の数百万円のうち数十万円が代行業者に消える。

AI-native に作り直して価格を一桁以上下げる余地がある領域。

## なぜ「Company Brain」へ拡張できるのか

補助金申請を解くアーキテクチャは次の 4 層構造：

1. **不変条件の構造化**（公募要領の経費区分・審査基準・文字数上限を Pydantic で型化）
2. **散在データの読込**（決算書・ヒアリング・過去申請書）
3. **LLM によるストーリー構築**（不変条件を満たしながら、事業者固有の物語を組む）
4. **品質チェック**（採択パターンと照合）

これは**そのまま**以下に拡張できる：

| 領域 | 不変条件 | 散在データ |
|---|---|---|
| 会計 | 複式簿記の貸借一致、税法、インボイス制度 | 領収書・請求書・通帳明細 |
| 資金繰り | 入出金タイミング、決済サイト、信用枠 | 通帳・売掛金・買掛金 |
| 労務 | 労基法、最低賃金、社会保険料率 | 勤怠・賃金台帳 |
| 経営判断 | KPI 閾値、競合動向、自社ベンチマーク | 売上明細・顧客リスト・SNS 反応 |

LLM と UI を対等な一級クライアントとして扱い、ビジネスロジック付き API を
本体とする設計は、補助金以外の領域にもそのまま転用できる。

## 設計上の判断

| 判断 | 理由 |
|---|---|
| 既存 SaaS の改修ではなく外から作る | 既存企業は内部複雑度のためフルリビルドのコストが見合わない |
| 人月モデルを採らず、定額・自走できるツールとして出す | 中抜きを再生産しないため |
| 散在データはローカル FS に正規化して持つ | クリーンな API を前提とする SaaS は中小企業の現実に届かない |
| 自己改善層を最初から組み込む | 案件ごとに使い捨てではなく、累積で賢くなる必要がある |

## なぜ自分が作るのか

私（池田計画合同会社 / 池田哲郎）は補助金申請コンサル現役で、毎日この
無駄を見ている。

エンジニア出身ではないが、Claude Code を相棒にすでに 16 体のエージェント
体制で実案件を回しており、複数の補助金で採択を取っている。仕様策定と
コードレビューは人間が、実装は AI が担当する協働開発体制を実地でやって
きた。

エンジニア視点だけでは「中小企業データの汚さ」が見えない。コンサル視点
だけでは「AI-native API 中心アーキテクチャ」に到達しない。**この交点は
希少**である。だから自分で作る。

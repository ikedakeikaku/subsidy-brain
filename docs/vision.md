# Vision — A Company Brain for SMBs

## 出発点：補助金は最も定型化された知的労働

日本の中小企業向け補助金は、典型的な「人月で稼ぐITコンサルが中抜きしている市場」（中島聡氏 2025-05-27号）の一例です。

- 公募要領（30〜60ページのPDF）を読む
- 事業者へヒアリングする
- 経費計画を組む
- 申請書を書く
- 採択基準と照らして自己採点する
- 不採択時に再申請する

このどれも、**ルールが明文化されている定型作業**です。一方で報酬は「成功報酬10〜20%」で、補助金の数百万円のうち数十万円が代行業者に消えます。これは中島氏のいう「中抜きの典型」そのもの。

AI-nativeに作り直せる代表例です。

## なぜ「Company Brain」へ拡張できるのか

補助金申請を解くアーキテクチャは：

1. **不変条件の構造化**（公募要領の経費区分・審査基準・文字数上限をPydanticで型化）
2. **散在データの読込**（決算書・ヒアリング・過去申請書）
3. **LLMによるストーリー構築**（不変条件を満たしながら、事業者固有の物語を組む）
4. **品質チェック**（採択パターンと照合）

これは**そのまま**以下に拡張できます：

| 領域 | 不変条件 | 散在データ |
|---|---|---|
| 会計 | 複式簿記の貸借一致、税法、インボイス制度 | 領収書・請求書・通帳明細 |
| 資金繰り | 入出金タイミング、決済サイト、信用枠 | 通帳・売掛金・買掛金 |
| 労務 | 労基法、最低賃金、社会保険料率 | 勤怠・賃金台帳 |
| 経営判断 | KPI閾値、競合動向、自社ベンチマーク | 売上明細・顧客リスト・SNS反応 |

中島氏が2026-05-19号で「AI-nativeビジネスアプリ・アーキテクチャ」として明確に書いた構造です。LLMもUIも対等な一級クライアントで、ビジネスロジックがAPIとして本体。

## 中島聡氏のメタトレンド観との整合

このプロジェクトは中島氏が直近1年で繰り返し書いているテーゼに乗っています：

- **「AIスーパー番頭構想」**（2024-12-24）— 会社のあらゆるデータにAIがアクセスし、知的労働の大半を任せる
- **「AIXは既存企業では起こせない」**（2025-04-01）— freee や 既存補助金コンサル会社からは出ない
- **「淘汰されるITコンサル」**（2025-05-27）— 人月モデルは破綻、補助金代行も同じ
- **「会計ソフトを2-3日で作った話」**（2026-05-12）— Claude Code × ビジネスロジックAPI × ファイルシステムDB
- **「SaaS終わりの始まり」**（2026-02-10）— ビジネスロジックの上のUIは AI-native では不要

## なぜ自分が作るのか

私（池田計画合同会社 / 池田哲郎）は補助金申請コンサル現役で、毎日この無駄を見ています。

エンジニア出身ではないが、Claude Code を相棒にすでに16体のエージェント体制で実案件を回しており、複数の補助金で採択を取っています。中島氏のいう「Vibe Coding を超えた、AIとペアプロするエンジニアリング」（2025-09-02号）を実地でやってきました。

エンジニア視点だけでは「中小企業データの汚さ」が見えません。コンサル視点だけでは「AI-native API中心アーキテクチャ」に到達しません。**この交点は希少**です。だから自分で作ります。
